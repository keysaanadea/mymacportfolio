Real-time Flow

Bagaimana Real-time Sync Bekerja

Setiap aksi pengguna pada canvas — memindahkan node, mengubah koneksi, mengedit label — langsung dikirim ke WebSocket server. Server mem-broadcast perubahan ke semua client yang terhubung ke topologi yang sama.


Optimistic UI — perubahan langsung terlihat di browser pengguna sebelum konfirmasi server
Room-based messaging — broadcast hanya ke pengguna pada topologi yang sama
Reconnect handling — client otomatis reconnect jika koneksi terputus
Auto-save <1s — perubahan dipersist ke MySQL segera setelah broadcast
Event deduplication mencegah double-processing pada rapid input

User Action

Engineer memindahkan node di canvas → event dikirim via WebSocket dengan payload minimal (node ID + posisi baru)

Server Broadcast

Node.js server menerima event, memvalidasi, lalu mem-broadcast ke semua client di room topologi yang sama — latency <50ms

Persist to DB

Laravel API menyimpan perubahan ke MySQL via Eloquent ORM — dipicu oleh WebSocket server setelah broadcast sukses

Canvas Implementation

Topology Canvas — Implementasi

Canvas topologi adalah komponen React yang paling kompleks dalam platform ini. Dibangun menggunakan Canvas API HTML5 dengan layer rendering terpisah untuk node, edge, dan overlay interaktif.


Double-buffered rendering — smooth 60fps bahkan dengan 100+ node
Viewport culling — hanya render node yang terlihat di viewport
Auto-layout engine — force-directed algorithm untuk posisi awal node
Snap to grid — node tertarik ke grid saat didrop untuk kerapian layout
Edge routing — garis koneksi otomatis menghindari node lain
Rendering Pipeline
1. Calculate viewport bounds
2. Cull off-screen nodes & edges
3. Render edges on back layer
4. Render nodes on front layer
5. Render interaction overlay (hover, selection)
6. requestAnimationFrame loop
Performance Target

60fps rendering dengan 100+ node & 200+ edge aktif di canvas. Viewport culling memotong render cost hingga 70% pada topologi besar.

Key Learnings

Pembelajaran & Tantangan

WebSocket at Scale

Room-based broadcasting mencegah message flooding ke client yang tidak relevan
Payload minimization — hanya kirim delta, bukan full state, untuk performa optimal
Heartbeat mechanism untuk mendeteksi dan memulihkan koneksi zombie

Canvas Performance

Viewport culling adalah optimisasi terpenting — impact langsung pada frame rate
requestAnimationFrame loop lebih efisien dari setInterval untuk rendering
Debounce drag events sebelum dikirim ke WebSocket mencegah flooding

Multi-user Collaboration

Optimistic UI dengan rollback saat server menolak perubahan
Last-write-wins strategy untuk conflict resolution pada edit bersamaan
Visual indicator untuk menunjukkan siapa sedang aktif di topologi yang sama

Enterprise Context

Bekerja langsung di lingkungan production enterprise dengan real users
Memahami kebutuhan monitoring jaringan dari perspektif operasional ISP
Pentingnya backwards compatibility saat merilis fitur baru ke production
