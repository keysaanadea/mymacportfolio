Overview

Project Overview

Project ini adalah Enterprise Knowledge Assistant yang saya bangun untuk PT Semen Indonesia Group agar karyawan dapat mengakses informasi perusahaan secara instan, baik lewat teks maupun suara, dengan konteks user dan session yang tetap terjaga.


Sistem memiliki routing bertingkat: business classifier di pintu depan, lalu route SOP, HC analytics, gabungan SOP+data, atau HC policy advisory sesuai intent dan role pengguna.

Lapisan Konteks

SOP & Kebijakan

Jawaban akurat dari dokumen PDF perusahaan menggunakan RAG hybrid, reranking, query analysis, dan policy injection untuk kasus aturan spesifik seperti perjalanan dinas

HC Analytics

Pertanyaan bahasa natural dikonversi ke SQL, divalidasi, dieksekusi ke database, lalu diperkaya dengan insight, narasi, dan rekomendasi visualisasi interaktif

HC Advisory

Route khusus Human Capital untuk evaluasi, harmonisasi, dan drafting kebijakan baru terhadap aturan existing serta data aktual

Core Features

Dua Mode Interaksi

Chat Mode

Antarmuka chat teks real-time dengan streaming SSE

  • Streaming jawaban token per token
  • Tabel data HC interaktif + Chart.js
  • Riwayat sesi multi-percakapan + restore context
  • Feedback thumbs up/down per jawaban
  • Role-based access (karyawan vs HC)
  • SINTA session bootstrap + user context injection
  • Markdown rendering otomatis

Call Mode

Overlay telepon full-screen — seperti menelepon AI

  • Rekam suara via MediaRecorder API
  • Transkripsi real-time (OpenAI Whisper)
  • AI bicara balik via ElevenLabs atau OpenAI TTS fallback
  • Loop: dengar → proses → jawab → ulang
  • Transcript muncul di layar
  • Jawaban dipadatkan untuk pengalaman call yang lebih natural
Features

Fitur Utama

RAG Hybrid v7.2.0

Pinecone vector search (top-15 chunks) + Cohere reranking multilingual untuk retrieval SOP yang akurat

Text-to-SQL Guarded

Bahasa natural → SQL → validasi keamanan → eksekusi PostgreSQL, dengan aturan SELECT-only, anti-injection, dan join registry

Travel Calculator

Kalkulasi biaya perjalanan dinas via Google Maps API — otomatis sesuai kebijakan perusahaan

8 Tipe Visualisasi

Bar, line, pie, doughnut, radar, dan lainnya — jenis grafik direkomendasikan AI, dirender Chart.js

Agentic HC Analyst

Pertanyaan HC kompleks dipecah menjadi plan bertahap, dievaluasi per langkah, lalu disintesis menjadi executive summary dan rekomendasi

SSE Streaming

Server-Sent Events untuk streaming jawaban real-time — dengan pembatalan request otomatis jika user pindah

Hybrid Memory

Redis hot cache 24 jam + Supabase persistent storage untuk riwayat percakapan yang cepat dan tahan lama

LLM-as-a-Judge

Background task mengevaluasi Faithfulness, Answer Relevance, dan Context Precision setiap interaksi

Langfuse Tracing

OTel-compatible observability — setiap request di-trace lengkap dari intent klasifikasi sampai jawaban akhir

Schema Explorer & Playground

HC dapat menelusuri schema HC, preview query, dan menyiapkan dataset low-code tanpa keluar dari antarmuka utama

Rate Limiting

SlowAPI per-IP: 15 req/menit untuk chat, 5 req/menit untuk call mode — mencegah penyalahgunaan API

Architecture

Alur Request — Chat Mode

E1 User / Browser Kirim pertanyaan via /ask atau /ask/stream
→ request ────►
P1 API Gateway Nginx + FastAPI · cek rate limit · siapkan trace · load history sesi
──────────────►
P2 Unified Classifier Klasifikasi intent: greeting / casual_chat / business
↓ greeting / casual
P2a Template Response Jawab langsung tanpa masuk engine · langsung balik ke browser
↓ business query — lanjut ke pipeline
P3 Role & Session Context Gate Pulihkan user_context + history aktif · cek role hasil SINTA · tentukan role efektif user
↔
D1 · Hybrid Memory
Redis — hot cache 24h Supabase — persistent history & session
↓ tentukan lane berdasarkan role user
P4 · Lane Non-HC / Karyawan A-only Fast Path
  • Query dikontekstualisasi dengan smart contextualize
  • Tidak memanggil classifier HC — request dipaksa ke Intent A / SOP retrieval only
  • Output bisa langsung balik sebagai jawaban SOP grounded ke dokumen sumber
  • Tidak boleh akses DB HC atau policy advisory
P5 · Lane HC HC Route Orchestrator
  • Standalone question dibentuk dulu dari history + context aktif
  • classify_hc_route() memilih A, B, AB, atau C
  • A: pertanyaan aturan → SOP engine
  • B: data → SQL gen → validator → executor → analyzer
  • AB: jalankan A & B paralel, lalu sintesis hasilnya
  • C: advisory kebijakan HC berbasis SOP existing sebagai source of truth
↓ akses knowledge / data store sesuai route
D2 SOP Knowledge Store Pinecone vector search · Cohere reranking multilingual · policy injection. Dipakai Route A, Route C, dan sisi SOP dari AB
D3 HC Data Store Supabase PostgreSQL · Text-to-SQL guarded · query analytics & visualisasi. Dipakai Route B dan sisi data dari AB
↓ hasilkan & persist jawaban akhir
P6 Final Response Assembly A-only atau B-only bisa langsung dirakit dan dikirim. Jika AB: sintesis SOP + data, lalu persist ke hybrid memory dan stream ke frontend  ·  Bg: Langfuse trace · LLM-Judge · feedback trace ID
↓ kirim jawaban ke user
E2 Browser Render Teks markdown · Chart interaktif (Chart.js) · Audio stream (call mode)
Auth SINTA saat ini menghasilkan dua role inti: karyawan dan hc. Karyawan selalu A-only, sedangkan HC dapat masuk ke route A, B, AB, atau C.
Route Features

Fitur per Route

Route A — SOP & Kebijakan

Pertanyaan aturan, prosedur, hak, tarif, dan dokumen

Query dianalisis dulu agar pencarian dokumen lebih tajam
Vector retrieval + multilingual reranking untuk memilih chunk paling relevan
Bisa menyisipkan aturan spesifik seperti perjalanan dinas dan kalkulasi rute
Output utama berupa jawaban ter-grounded ke dokumen sumber

Route B — HC Analytics

Pertanyaan faktual dan analitis ke database HC

Natural language diubah ke SQL PostgreSQL yang sudah dibatasi dan divalidasi
Ada guard untuk JOIN resmi, grain data, dan deduplikasi headcount
Hasil diperkaya narasi, insight, SQL transparency, dan chart recommendation
Pertanyaan kompleks bisa naik ke mode agentic analyst multi-step

Route AB — SOP + Data

Pertanyaan yang butuh aturan dan data aktual sekaligus

Menjalankan route A dan B secara paralel untuk menghemat waktu
Cocok untuk simulasi biaya, impact policy, atau evaluasi aturan berbasis data
Tahap akhir melakukan synthesis agar hasil SOP dan data tidak terpisah
Output menekankan aturan, data faktual, simulasi, dan rujukan dokumen

Route C — HC Policy Advisory

Khusus HC untuk merancang atau meninjau kebijakan

Memeriksa dukungan, konflik, gap, dan kebutuhan revisi pada aturan existing
Reuse retrieval SOP sebagai sumber kebenaran utama
Bisa dipakai untuk draft policy baru, harmonisasi aturan, atau evaluasi kebijakan
Jika perlu bukti data aktual, pertanyaan bisa diarahkan ke AB
Playground

Playground Bisa Apa Saja?

Playground adalah workspace low-code untuk tim HC menyiapkan query dan dataset HC tanpa harus menulis SQL mentah dari nol.


Mode ini mengambil schema HC, known joins, dan statistik kolom untuk membantu user membangun transformasi data secara visual atau dengan bantuan AI builder.

Catalog Live Preview Shared Dataset AI Builder

Catalog & Schema

Melihat daftar tabel HC, kolom yang tersedia, tipe data, serta relasi join yang sudah diketahui sistem.

AI Build

User bisa menulis permintaan natural seperti "total lembur per direktorat", lalu AI mengubahnya menjadi PlaygroundSpec JSON yang siap dipreview.

Live Preview & Publish

Spec divalidasi, diubah menjadi SQL, dijalankan dalam mode preview, lalu bisa disimpan sebagai session atau dipublish menjadi shared dataset.

Filter Tree

Menyusun filter sederhana maupun kombinasi AND/OR untuk slicing data yang lebih presisi.

Known Join Guidance

Playground mengarahkan user memakai relasi tabel resmi agar query tidak asal menggabungkan data.

Metrics & Grouping

Mendukung agregasi seperti sum, avg, count, count_distinct, sorting, ranking, dan limit hasil.

Session & Shared View

Hasil kerja bisa disimpan sebagai session internal atau dipublish sebagai shared view untuk organisasi.

Quick Export

Preview hasil dapat diekspor cepat ke CSV dari antarmuka frontend untuk analisis lanjutan.

Validation Layer

Spec dan SQL dicek dulu sebelum dijalankan agar tetap aman dan sesuai schema HC yang valid.
